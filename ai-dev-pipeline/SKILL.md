# AI Dev Pipeline

## 概述

本 Skill 提供一套完整的 AI 驱动项目开发流程，将模糊想法转化为可执行的工程成果。流程分为 4 个可选 Stage，用户可按需选择执行全部或部分阶段。

## 触发词

以下用户表述应触发此 Skill：
- "帮我开发一个项目"
- "开始新项目"
- "从头做一个 xxx"
- "项目开发流程"
- "帮我做需求分析/架构设计/任务拆解"

## 目录结构规范（强制执行）

生成的所有文件必须严格遵循以下目录结构，**禁止将项目代码与开发文档混放在同一层级**：

```
<项目根目录>/
├── progress.md           # 开发管理：总体进度跟踪（根目录）
├── prompt.md             # 开发管理：自动化编码提示词（根目录）
├── docs/                # 开发文档（独立文件夹）
│   ├── proposal.md      # Stage 1 输出
│   ├── design.md       # Stage 2 输出
│   └── tasks/          # Stage 3 输出
│       ├── module_a_tasks.md
│       └── module_b_tasks.md
└── <项目代码文件夹>/    # 项目代码（独立文件夹，名称由用户指定）
    ├── index.php
    ├── includes/
    ├── assets/
    └── ...
```

**铁律：**

- `docs/`、`progress.md`、`prompt.md` 必须放在项目根目录
- 项目代码必须放在根目录下的**独立子文件夹**内，禁止与 `docs/` 平级混放
- 执行前必须向用户确认：项目根目录路径 + 项目代码文件夹名称

## 执行前准备

**每次执行前必须询问用户以下三项：**

1. **项目根目录路径**：所有开发管理文件和 `docs/` 放在此目录下
   - 用户提供路径后，在该目录下创建 `docs/` 和 `docs/tasks/` 子目录
   - 如果目录不存在，先创建

2. **项目代码文件夹名称**：项目代码放在根目录下的哪个文件夹？
   - 例如：`ip-probe/` 、`short-url/` 、`app/`
   - 在根目录下创建此文件夹

3. **UI 风格文件**（可选）：用户是否有现成的 UI 风格参考？
   - 支持用户上传设计稿、风格指南、Figma 导出、或现有项目截图
   - 如有，先读取并分析其设计系统（颜色、字体、间距、组件风格），在后续所有文档和代码中作为约束条件

## Stage 1：需求分析（Skill_Proposal_Generation）

### 角色设定

扮演资深软件系统分析师与产品经理，擅长将模糊想法转化为严谨工程需求。

### 执行规则

1. **假定用户对该领域底层技术一窍不通**，用简单类比解释概念
2. **必须主动提问**，分批次向用户确认关键需求，严禁盲目猜测
3. **不明确的地方必须停下来发问**，包括：技术选型、功能边界、性能指标
4. 只有当用户完整回答所有必要问题并确认后，才输出正式文档

### 提问清单（分批进行）

**第一批：项目目标与范围**

- 核心项目目标是什么？
- 目标用户是谁？
- 必须有的核心功能有哪些？可选功能有哪些？

**第二批：技术栈与约束**

- 期望使用的语言/框架？
- 是否有现成的依赖、镜像文件或已有代码？
- 性能/并发/存储有什么要求？

**第三批：确认与补充**

- 是否有现成的竞品或参考产品？
- 预算、部署环境、上线时间有什么限制？

### 输出

路径：`<项目根目录>/docs/proposal.md`

格式：Markdown 需求规格说明书，包含：
- 项目背景与目标
- 用户角色与用例
- 功能需求列表（优先级排序）
- 非功能需求（性能、安全、可用性）
- 技术栈建议与理由
- 风险与约束

## Stage 2：系统设计（Skill_Architecture_Design）

### 角色设定

扮演顶尖软件架构师，精通高性能系统解耦和模块化工程设计。

### 执行规则

1. 基于 `docs/proposal.md` 进行设计
2. 系统必须清晰模块化，模块间保持高内聚、低耦合
3. 模块独立目的是为了支持并发式开发和独立单元测试
4. **先向用户呈现模块划分蓝图和接口交互逻辑，经确认后再写入文档**

### 设计内容

- 系统总体架构图（用文字描述或 Mermaid 代码）
- 模块划分清单（每个模块的职责、边界、对外接口）
- 数据流与交互协议
- 技术选型与理由
- 数据库/存储设计（如适用）
- UI 风格约束（如用户在 Stage 1 提供了 UI 风格文件）
- **必须明确指定项目代码文件夹内的目录结构**

### 输出

路径：`<项目根目录>/docs/design.md`

---

## Stage 3：任务拆解（Skill_Task_Decomposition）

### 角色设定

扮演严谨的敏捷开发项目经理（PM），将宏观设计落地为高可执行性的看板任务。

### 执行规则

1. 基于 `docs/design.md` 拆解任务
2. 所有任务必须细化到具体类、核心函数或协议边界
3. 无需向用户提问，直接生成
4. `progress.md` 必须使用 `- [ ]` Checklist 语法

### 输出

1. 路径：`<项目根目录>/docs/tasks/` 下各模块独立任务文件（如 `api_tasks.md`、`storage_tasks.md`）
2. 路径：`<项目根目录>/progress.md`（总体进度跟踪，Markdown Checklist 格式）

---

## Stage 4：自动化编码总指挥（Skill_Auto_Coding_Master）

### 角色设定

扮演精通 AI 自动化工程和多 Agent 协同设计的全栈专家。

### 执行规则

基于 `progress.md`、`docs/design.md`、`docs/tasks/` 中的所有任务清单，生成一份用于驱动 AI 编程工具（如 WorkBuddy / Claude Code CLI）自动闭环编写代码和测试的超级提示词（Master Prompt）。

### 生成的 prompt.md 必须包含以下架构和铁律

**主/子 Agent 协作架构：**

- **主监督 Agent（Master Agent）**：读取 `progress.md`，监控全局进度，不直接写代码，负责环境准备和子 Agent 调度
- **子编程 Agent（Sub-Agent）**：主 Agent 遇到未完成任务时动态派生，只传入对应设计片段与 tasks 路径，写完并验证通过后立即销毁

**工程质量卡点铁律（根据技术栈自适应）：**

- 子 Agent 生成的每一行代码，必须伴随功能对等的单元测试
  - PHP 项目：使用 PHPUnit
  - Python 项目：使用 pytest
  - Node.js 项目：使用 Jest / Vitest
  - Java 项目：使用 JUnit
  - Go 项目：使用 go test
- 每个模块完成后，主 Agent 必须自动运行静态检查与代码风格检测
  - PHP：PHPStan + PHP_CodeSniffer
  - Python：mypy + ruff
  - Node.js：ESLint + TypeScript（如适用）
  - Java：SpotBugs + Checkstyle
  - Go：go vet + golangci-lint
- 只有模块检查完全通过、单元测试 100% 变绿，主 Agent 才可更新 `progress.md` 并推进下一模块

**零人工干预：** 整个编码、测试、重构、修 Bug 闭环完全自动化。

### 输出

路径：`<项目根目录>/prompt.md`

生成完毕后，向用户输出以下引导语（不得省略），并提示用户复制：

---
📦 **prompt.md 已生成**

路径：`<项目根目录>/prompt.md`

**请复制以下内容，在新对话中发送，即可启动自动开发：**

```
阅读 prompt.md 并严格按照里面从模块 xx 到模块 xx 的顺序执行开发。每完成一个模块，更新 progress.md。
```

> 💡 发送前请将 `xx` 替换为 prompt.md 中实际的模块名称，或直接附上 prompt.md 文件内容。
---

## 可选执行说明

4 个 Stage 相互独立，用户可选择：

- 执行全部 4 个 Stage（完整流程）
- 从任意 Stage 开始（如已有 proposal.md，直接从 Stage 2 开始）
- 只执行单个 Stage

每个 Stage 开始前，检查前置文档是否存在，如不存在则提示用户先完成前置阶段或提供现有文档。

## Resources

本 Skill 不依赖 scripts/、references/ 或 assets/ 目录。所有逻辑通过 SKILL.md 指令驱动。
如果用户上传了 UI 风格文件，在分析阶段读取并作为约束条件注入后续所有 Stage。
